## 2. 任务 D2-1：谜题系统 (L1 + L2)

### 2.1 背景

`docs/dev-plan-full.md` §2.4 + §3.4 已定义：

- **L1 = 观察型**（50%）：颜色匹配 / 文字顺序，不需"道具栏"
- **L2 = 收集型**（30%）：散落物 / 简单组合，需要物品栏
- **L3 = 序列型**（20%，仅 Ch.3 用）— **本 sprint 不做**
- 统一接口在 `src/puzzle/`，3 个通用 solver

**D2-1 范围**：L1 + L2 引擎 + 集成到 PROLOGUE / CHAPTER_X 状态机。sprint-week1 实际只需要 **4 个 puzzle**：
- `prologue_p1_lantern_color` (L1，颜色序列)
- `prologue_p2_postcard_order` (L1，文字顺序)
- `prologue_p3_signal_wire` (L1，颜色序列变体)
- `ch1_p1_antler_collect` (L2，鹿骨收集)

后续 Ch.1 灯笼 L1 在 D3+ 加。L3 整个 sprint 不写。

### 2.2 必做步骤（估时 2h）

1. **建目录结构**（~5 min）
   ```
   src/puzzle/
     index.ts          # 统一导出
     types.ts          # Puzzle<L>, SolveResult, Solution<L>
     registry.ts       # registerPuzzle, trySolve
     bootstrap.ts      # 启动时注册 4 个 sprint puzzle
     solvers/
       colorSequence.ts    # L1
       collectItems.ts     # L2
     ui/
       colorPicker.ts      # 4 色弹窗（红/白/蓝/黄，tokyo_heisei 调色板）
     __tests__/
       colorSequence.test.ts
       collectItems.test.ts
   src/ui/inventory.ts  # 5×2 网格，32×32 图标
   ```

2. **类型定义**（~10 min，参考 dev-plan-full.md §3.4）
   ```ts
   export type PuzzleLevel = 1 | 2 | 3;
   export interface Puzzle<L extends PuzzleLevel> {
     id: string;
     chapter: 1 | 2 | 3;
     type: 'L1' | 'L2' | 'L3';
     hint: string;
     solution: Solution<L>;
     onSolve(): SceneTransition | null;
   }
   export type SolveResult =
     | { ok: true; nextScene: SceneTransition | null }
     | { ok: false; reason: 'wrong' | 'incomplete' | 'invalid-state' };
   ```

3. **L1 colorSequence solver**（~25 min）
   - 输入：`string[]` 用户选的颜色序列
   - 校验：跟 `solution.sequence` 完全一致
   - 配套：`src/puzzle/ui/colorPicker.ts` 弹窗 4 色
   - **vitest 单测 ≥ 5 case**：正例 / 反例 / 部分匹配 / 空 / 越界

4. **L2 collectItems solver**（~25 min）
   - 输入：`string[]` 当前物品栏 ID
   - 校验：跟 `solution.required` 完全匹配
   - 配套：`src/ui/inventory.ts` 5×2 网格
   - **vitest 单测 ≥ 4 case**：够 / 不够 / 多 1 个 / 空

5. **registry + bootstrap**（~10 min）
   - `registerPuzzle(p)` 存到 `Map<id, Puzzle>`
   - `trySolve(id, input)` 路由到对应 solver
   - `bootstrap()` 注册 4 个 sprint puzzle（hardcode 数据，写 TODO 注释说明 sprint-week1 末期要搬 yaml）

6. **状态机集成**（~25 min）
   - **不动 6 态主状态机**（INTRO/MENU/.../ENDING）
   - 只在 `src/renderer/src/states/CHAPTER_X.ts` 子状态机加 puzzle 流程
   - 流程：scene 节点标 `puzzles: [{ id, ... }]` → state 读到 → 弹 puzzle UI → 用户操作 → `trySolve` → `ok` 跳 nextScene / `fail` 重提示
   - 用简单 marker 触发（如 `dialog: "…[puzzle:prologue_p1_lantern_color]"`），**不**引入新 DSL

7. **demo 验证**（~10 min）
   - 改 `assets/text/demo-script.json` 加 1 个 L1 puzzle 节点（ch1_p1_antler_collect L2 也加 1 个，保险起见）
   - `pnpm dev` 跑通，手动点几下确认能 solve

8. **跑门禁**

## 3. 已知陷阱（CLAUDE.md 22+ 条必读）

完整列表见工作目录的 `CLAUDE.md`。**重点关注**：

- **pnpm only**（不要 npm/yarn）
- **相对路径 off-by-one**：`src/renderer/src/puzzle/...` 到 `assets/` 是 `../../../../assets/...`（**5 层**！D1 字体是 4 层，puzzle 在 renderer/src 下又深一层）；`src/puzzle/...`（在 `src/` 下，**不在 renderer/src 下**）到 `assets/` 是 `../../assets/...`（**2 层**）。**分清楚再写路径**。
- web tsconfig 不含 Node 类型 → renderer 那边的代码别 `import fs`
- `import x from './foo.json'` 在 web tsconfig 下需要 `.d.ts` 声明
- `Proxy({} as T, handler)` 必须 `as unknown as T`
- biome `lint/style/noCommaOperator` / `lint/complexity/useLiteralKeys`
- `.claude/**` 加到 `biome.json#files.ignore`
- **vitest happy-dom 15 不实现 `CanvasRenderingContext2D`** → Proxy mock
- 测试 mock 暴露的 `__press` / `__pressed`，`ctx.input` 要扩成 `MockInput`
- 对话/动画测试要"按帧推进"
- **electron-builder 24 Windows symlink**：开发模式可以，但生产 pack 要 Developer Mode
- PowerShell heredoc `"` 行被解析成 arg（用 %TEMP% + git commit -F 绕）
- `.git/` 路径被 safety hook 拦（走 git CLI 不直读路径）
