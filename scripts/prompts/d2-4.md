## 2. 任务 D2-4：闪退保护

### 2.1 背景

`docs/dev-plan-full.md` §3.6 已定义：

> **防闪退保护**（已在一周冲刺 D2）：所有 `transition()` 包 try/catch + 写 `crash-recovery.json`

Electron 主进程 / 渲染进程都可能在玩家不知情时崩。**目标**：崩了下次启动能恢复到崩前 1 个状态。

### 2.2 必做步骤（估时 2h）

1. **建目录结构**（~5 min）
   ```
   src/main/
     crash-recovery.ts   # 主进程: 写/读 userData/crash-recovery.json
     window.ts           # 现有, 包 try/catch
   src/preload/
     recovery-api.ts     # 暴露给渲染进程: getRecovery / clearRecovery
   src/renderer/src/
     boot-guard.ts       # 启动时检查 recovery, 弹"上次崩了, 恢复?"对话框
   src/main/__tests__/
     crash-recovery.test.ts
   ```

2. **crash-recovery.ts**（主进程, ~30 min）
   - 写：
     ```ts
     type RecoveryState = {
       lastSafeState: GameStateSnapshot;  // 见 src/shared/types.ts
       lastTransitionAt: number;          // unix ms
       crashCount: number;                // 连续崩次数
     };
     function writeRecovery(state: GameStateSnapshot): Promise<void>;
     function readRecovery(): Promise<RecoveryState | null>;
     function clearRecovery(): Promise<void>;
     ```
   - 路径：`app.getPath('userData') + '/crash-recovery.json'`
   - 写入用 `fs.promises.writeFile` + 临时文件 + rename（**原子写**，避免半写状态）
   - **每次成功 `transition()` 之后**调用 `writeRecovery`，作为"最近一次安全状态"

3. **transition() try/catch 包装**（~25 min）
   - 找 `src/renderer/src/states/` 里的所有 `transition()` 调用点
   - 包 try/catch：
     ```ts
     try {
       await transition(newState);
       await window.recovery.write(snapshot(newState));
     } catch (err) {
       console.error('[transition failed]', err);
       // 不弹 UI, 让 Electron 默认 error dialog 处理
       // 渲染进程崩了主进程也知道（preload 的 onerror）
       throw err;
     }
     ```
   - 关键：`writeRecovery` **必须**在 transition 成功之后（不然存的是"半成功"状态）

4. **preload 暴露 API**（~10 min）
   - `src/preload/recovery-api.ts`:
     ```ts
     contextBridge.exposeInMainWorld('recovery', {
       write: (s) => ipcRenderer.invoke('recovery:write', s),
       read: () => ipcRenderer.invoke('recovery:read'),
       clear: () => ipcRenderer.invoke('recovery:clear'),
     });
     ```
   - `src/shared/api.ts` 加 `RecoveryApi` interface

5. **boot-guard.ts**（渲染进程, ~20 min）
   - 启动时（在 `main.ts` 早期）调 `window.recovery.read()`
   - 如果有 recovery 且 `Date.now() - lastTransitionAt < 5min`（**5 分钟内**才恢复，太旧了不恢复）：弹"上次游戏未正常结束（X 分钟前），是否恢复？"
   - 用户选"恢复" → `state.load(snapshot)` → 跳到 lastSafeState 对应 scene
   - 用户选"不恢复" → `clearRecovery()` → 走正常 INTRO
   - **5 分钟阈值**是 dev-plan §3.6 暗示的（自动存档带轮转），写代码时确认 OR 调成 30 分钟（半章时长）

6. **crash detection**（~15 min）
   - 主进程：`process.on('uncaughtException')` + `process.on('unhandledRejection')` → 写 recovery 后再退出
   - 渲染进程：监听 `window.onerror` / `window.onunhandledrejection` → IPC 通知主进程写 recovery
   - **不要**让这些 handler 阻止进程退出（让 Electron 该崩就崩，崩完才有 recovery 可读）

7. **单测**（~15 min）
   - `crash-recovery.test.ts`：
     - `writeRecovery` 后 `readRecovery` 拿回相同数据
     - 原子写测试：写入过程中 kill 进程（mock fs.writeFile throw）不留下半文件
     - `clearRecovery` 后 `readRecovery` 返回 null
     - crashCount 自增测试：连续 2 次 write recovery → 第二次 read 出 crashCount=1

8. **跑门禁**
   ```bash
   pnpm typecheck
   pnpm test
   pnpm exec biome format --write .
   pnpm build
   ```

9. **Commit（1 个）**
   - 标题：`feat: 闪退保护 try/catch + crash-recovery.json (D2-4)`
   - body 列改的文件树 + 4 个恢复点（transition / uncaughtException / unhandledRejection / boot-guard）

## 3. 已知陷阱（CLAUDE.md 22+ 条必读）

- **vitest happy-dom 15 不实现 `CanvasRenderingContext2D`** → Proxy mock（你主进程单测不涉及 canvas，但 preload 单测可能 mock window）
- **electron-builder 24 Windows symlink**
